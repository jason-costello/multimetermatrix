    The problem: The legend bar and table header both scroll away when scrolling the page vertically. The legend bar has no `position: sticky`. The header's `top: 0` is too aggressive — it clings to the viewport top edge, overlaying the fixed top bar.

    Root cause: The body is the vertical scroll container. `position: sticky; top: 0` on #table-header positions it 0px from the viewport top, UNDER the fixed #top-bar (48px height, z-index: 50). The legend bar has no sticky positioning at all.

    Fix approach: Keep body as scroll container. Add sticky positioning to the legend bar. Adjust the table header's sticky top offset to account for top bar + legend bar height. Use a JS helper to measure the legend bar height dynamically (since legend bar height depends on rendered content).

    **CSS changes to site/style.css:**

    1. Add sticky positioning to `#legend-bar` (section 4):
       Locate the existing `#legend-bar` rule block. Add three properties:
       ```css
       position: sticky;
       top: 48px;  /* below the fixed top bar */
       z-index: 8;
       ```
       Insert these within the existing rule, preserving all other properties (display, flex-direction, gap, padding, background, border-bottom).

    2. Change `#table-header` `top` value (section 5):
       Locate `#table-header { position: sticky; top: 0; z-index: 10; }`.
       Change `top: 0` to:
       ```css
       top: calc(48px + var(--legend-bar-height, 44px));
       ```
       This accounts for the fixed top bar (48px) plus the rendered height of the legend bar. The CSS variable is set by JS.

    3. Ensure `#legend-bar` has `background: var(--bg-legend)` (already does, but verify) so content doesn't show through when sticky.

    **JS changes to site/app.js:**

    4. Add an `updateStickyOffsets()` function after the `updateStickyBrandOffset()` function (around line 323):
       ```javascript
       // ===========================================================================
       // Sticky Legend/Header Offset
       // ===========================================================================

       /**
        * Measure the legend bar height and set --legend-bar-height CSS custom property
        * on the #table-header element for correct sticky positioning.
        */
       function updateStickyOffsets() {
         var legendBar = document.getElementById('legend-bar');
         var tableHeader = document.getElementById('table-header');
         if (legendBar && tableHeader) {
           var h = legendBar.offsetHeight;
           tableHeader.style.setProperty('--legend-bar-height', h + 'px');
         }
       }
       ```

    5. In `initUI()` (around line 178, after populating legend swatches and before "Render header row"), add a call to `updateStickyOffsets()`:

           var modelWidth = (el.tableHeader.firstElementChild && el.tableHeader.firstElementChild.offsetWidth) || 150;

<task type="auto">
  <name>Task 2: Fix hover highlight text invisibility</name>

  <files>
    site/style.css
  </files>

  <read_first>
    site/style.css (section 7 — Row Alternating + Hover, lines 256-270)
  </read_first>

  <action>
    The problem: `tr:hover td` sets `background: rgba(37,99,235,0.04) !important` which overrides cell-level band/flag background colors. For cells with white text (#ffffff) on dark colored backgrounds (green, red, orange, gray), the blue-tinted white hover background makes the white text invisible.

    Root cause: The `!important` flag on the hover background overrides inline styles set on `<td>` elements (which carry the band/flag color). The hover highlight replaces the colored background rather than overlaying on top of it.

    Fix: Replace `background` override with `box-shadow` overlay. An inset box-shadow with the same color overlays a tint on top of the existing cell background without replacing it, preserving the cell's background color and text contrast.

    **CSS changes to site/style.css (Section 7):**

    1. Locate the existing rule block:
       ```css
       tr:hover td {
         background: rgba(37, 99, 235, 0.04) !important;
         transition: background-color 0.1s ease;
       }
       ```

    2. Replace it with:
       ```css
       tr:hover td {
         box-shadow: inset 0 0 0 1000px rgba(37, 99, 235, 0.04);
         transition: box-shadow 0.1s ease;
       }
       ```

    The value `1000px` for the spread radius ensures the shadow covers the entire cell regardless of its size (since 1000px > any realistic cell height/width). The alpha channel (0.04 = 4%) provides the same subtle blue tint as before, but as an overlay on top of the existing background rather than a replacement.

    No other changes needed — the `tr { cursor: pointer }` rule, row alternation (`tr:nth-child(even) td`), and all other hover-related styles remain unchanged.
  </action>

  <verify>
    <automated>grep -c "box-shadow: inset" site/style.css | grep -q "1" && echo "box-shadow hover found"</automated>
    <note>Verify the background: !important no longer exists on tr:hover td. The box-shadow line must exist.</note>
  </verify>

  <done>
    - `tr:hover td` uses `box-shadow: inset ...` instead of `background: ... !important`
    - No `background` override with `!important` exists on `tr:hover td`
    - Band-colored cells (green, amber, red, orange, gray) show hover overlay without losing text visibility
    - Default cells (no band/flag) still show the blue tint on hover
  </done>
