  <action>
    Create `.github/workflows/refresh.yml` with the following structure. The `.github/workflows/` directory does not exist yet — create it.

    Workflow name: "Refresh Data"

    Triggers:
    - `schedule` with cron `0 6 * * 1` (Monday 06:00 UTC)
    - `workflow_dispatch` (manual trigger)

    Top-level permissions (set these at workflow level, they apply to all jobs):
    - `contents: write` (needed for git commit/push of data.json and for checkout)
    - `pages: write` (needed for deploy-pages action)
    - `id-token: write` (needed for deploy-pages OIDC)

    Concurrency:
    - Group: `pages`
    - `cancel-in-progress: false` per D-07

    Jobs:

    Job 1 — `build` (runs-on: ubuntu-latest):
    Steps:
    1. `actions/checkout@v4`
    2. `actions/setup-go@v5` with:
       - `go-version-file: go.mod` (reads go 1.25.0 from go.mod, no hardcoded version)
       - `cache: true` (built-in setup-go caching for Go module cache)
    3. `go run ./cmd/meters fetch` — downloads xlsx. Non-zero exit propagates to workflow failure.
    4. `go run ./cmd/meters build` — parses xlsx and writes data.json. Non-zero exit propagates to workflow failure.
    5. Conditional commit step — run via shell script:
       - Use `git diff --quiet data.json` to check if data.json changed.
       - If exit code != 0 (data changed), configure git user.name as `github-actions[bot]` and git user.email as `github-actions[bot]@users.noreply.github.com`, then `git add data.json`, `git commit -m "chore: refresh data.json"`, and `git push`.
       - If exit code == 0 (data unchanged), output an informational message like "data.json unchanged, skipping commit" — no error, no-op.
    6. Prepare deployment artifact directory:
       - `mkdir -p _site`
       - `cp index.html data.json _site/`
       - `cp -r site _site/` (copies site/ directory with app.js, engine.js, style.css)
       - `touch _site/.nojekyll` (generated in artifact per D-05 — the committed .nojekyll at repo root is handled in Task 2)
    7. `actions/upload-pages-artifact@v4` with `path: _site`

    Job 2 — `deploy` (runs-on: ubuntu-latest):
    - `needs: build` (automatically skipped if build job fails, satisfying D-07)
    - `environment:` name `github-pages`, url from `${{ steps.deployment.outputs.page_url }}`
    - Steps:
      1. `id: deployment`, uses `actions/deploy-pages@v4`

    Use exact action versions as specified above — do not use `@main` or `@master` or unversioned references. Only first-party `actions/*` actions, no third-party actions.

    The workflow must fail (non-zero exit) when `go run ./cmd/meters fetch` or `go run ./cmd/meters build` exit non-zero. The `deploy` job must be skipped in this case because it depends on `build` (not because of explicit condition checks — the `needs:` dependency handles this automatically per GitHub Actions semantics).
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| .github/workflows/refresh.yml | Workflow YAML is part of the repo; any change to the workflow must go through a PR (or direct push on main depending on branch protection). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | .github/workflows/refresh.yml action versions | mitigate | Pin all actions to specific major versions (actions/checkout@v4, actions/setup-go@v5, actions/upload-pages-artifact@v4, actions/deploy-pages@v4). No third-party actions used. Version pins prevent supply chain drift to unexpected major releases. |
| T-03-02 | Elevation of Privilege | GITHUB_TOKEN with contents:write | accept | Token is scoped to single public repo containing no secrets, PII, or sensitive data. Commit step only writes data.json. GitHub's built-in token scoping limits blast radius. |
| T-03-03 | Spoofing | Go module resolution in CI | mitigate | go.mod + go.sum pin exact module checksums. `go run ./cmd/meters fetch` verifies build integrity against go.sum before resolving modules. No transitive dependency surprise is possible. |
| T-03-04 | Denial of Service | Schedule trigger on high-frequency rerun | mitigate | concurrency group `pages` with `cancel-in-progress: false` prevents overlapping runs without canceling in-flight ones. A manual workflow_dispatch while a scheduled run is in progress will queue rather than disrupt. |
</threat_model>
