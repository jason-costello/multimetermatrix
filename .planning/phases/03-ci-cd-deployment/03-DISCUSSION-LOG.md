## data.json Git Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Stop committing data.json | Add to .gitignore, require local pipeline run. Cleaner git history. | |
| Keep committing for local dev | data.json stays committed for local dev convenience. CI generates fresh for deploy. | ✓ |

**User's choice:** Keep committing data.json for local dev. Frontend works immediately after git clone — no extra setup step required.

**Follow-up: CI commits on success?**

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, CI commits on success | After successful deploy, CI commits fresh data.json back to main. | ✓ |
| No, local dev updates manually | CI only generates for deploy artifact. Committed data.json is a snapshot. | |

**User's choice:** CI commits fresh data.json to main on success, keeping the committed copy in sync with live data. Existing `.github/workflows/refresh.yml` pattern from prior planning carries forward.

**Follow-up: conditional commit?**

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, conditional commit | CI checks `git diff --exit-code data.json` before committing. No change = no commit. | ✓ |
| No, always commit | Always commit on success, even when data hasn't changed. | |

**User's choice:** Conditional commit — avoid no-op commits when data is unchanged between weekly runs.

## Deploy Artifact File Set

| Option | Description | Selected |
|--------|-------------|----------|
| Root + site/ + .nojekyll | index.html, data.json (CI-generated), site/*, .nojekyll. Clean artifact with exactly what Pages needs. | ✓ |
| Everything at root + site/ | Without .nojekyll. Risk: Jekyll processing breaks static site. | |

**User's choice:** Artifact includes index.html, data.json, site/*, and .nojekyll. `.nojekyll` prevents Jekyll processing on GitHub Pages.

**Follow-up: commit .nojekyll to git?**

| Option | Description | Selected |
|--------|-------------|----------|
| Commit + CI generates | .nojekyll committed to repo AND generated by CI in artifact. Always present. | ✓ |
| CI generates only | Only CI creates .nojekyll in artifact. Not committed. | |

**User's choice:** Commit .nojekyll to git AND generate in CI. Redundant but harmless — ensures it's always present.

## GitHub Pages Setup

| Option | Description | Selected |
|--------|-------------|----------|
| Manual setup in Settings | Settings → Pages → Source = "GitHub Actions". One-time manual step. | ✓ |
| Auto via workflow | Workflow calls actions/configure-pages@v5 to set source programmatically. | |

**User's choice:** Manual one-time setup. Simpler workflow YAML — only needs actions/deploy-pages@v4. No configure-pages action dependency.

## CI Failure Behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, old site stays up | On failure: existing deployment untouched. Users see last known good data. | ✓ |
| No, take site down | On failure: remove deployment. Users see 404. | |

**User's choice:** Old site stays up on failure. Deploy step is skipped → no new artifact → GitHub Pages serves last successful deploy. Satisfies ROADMAP SC-5 ("stale data is never deployed").

**Follow-up: notification?**

| Option | Description | Selected |
|--------|-------------|----------|
| GitHub default is enough | Workflow failure → email to watchers + red X in Actions tab. | ✓ |
| Add explicit notification | Slack/Discord/Telegram webhook step on failure. | |

**User's choice:** GitHub default notification (email to repo watchers) is sufficient. No extra notification integration needed.
