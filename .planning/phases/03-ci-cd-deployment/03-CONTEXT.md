# Phase 03: CI/CD & Deployment - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

## Phase Boundary

GitHub Actions workflow that auto-refreshes data weekly (Monday 06:00 UTC) and deploys the static site to GitHub Pages via artifact-based upload. Workflow runs checkout → setup-go → fetch → build → deploy end-to-end. On success, conditionally commits fresh data.json back to main. On failure, old deployment stays up — stale data never deployed.

**In scope:** Weekly cron + workflow_dispatch trigger, Go pipeline execution (fetch + build), artifact-based Pages deployment (upload-pages-artifact + deploy-pages), .nojekyll generation, conditional data.json commit on success.

**Out of scope:** v2 features (URL state, compare mode, dark mode), branch preview deploys, custom domain setup, Slack/Discord notification integration, PR-based preview environments.

## Implementation Decisions

### data.json Git Strategy
- **D-01:** data.json stays committed to git for local dev convenience — `git clone` + open index.html works immediately
- **D-02:** CI commits fresh data.json to main on successful workflow run, keeping the committed copy in sync with live data
- **D-03:** Conditional commit — CI checks `git diff --exit-code data.json` before committing. No-op when data hasn't changed (avoids empty commit noise)

### Deploy Artifact
- **D-04:** Artifact includes exactly: `index.html`, `data.json` (CI-generated), `site/*` (app.js, engine.js, style.css), `.nojekyll`
- **D-05:** `.nojekyll` committed to git AND generated by CI in the deploy artifact

### GitHub Pages Configuration
- **D-06:** Manual one-time setup: repo Settings → Pages → Source = "GitHub Actions". Workflow uses `actions/deploy-pages@v4` (no configure-pages action needed)

### CI Failure Behavior
- **D-07:** On pipeline failure (fetch HTTP error, build validation error), the deploy step is skipped — existing GitHub Pages deployment stays up with last known good data
- **D-08:** GitHub default notification (email to repo watchers on workflow failure) is sufficient — no explicit notification integration needed

## Existing Code Insights

### Reusable Assets
- `cmd/meters/` — Go CLI with `fetch` and `build` subcommands. Exit codes: 0=success, 1=internal, 2=invalid usage, 3=fetch HTTP error, 4=parse error, 5=data validation error. CI must detect non-zero exit.
- `data.json` (992KB) — Currently committed, updated by CI on success
- `index.html` + `site/` — Static frontend files deployed to Pages

### Established Patterns
- **Zero-dependency** — No npm, no CDN, no build step. CI only needs Go.
- **Loud failure** — CLI exits non-zero on structural mismatches with clear stderr messages. CI propagates exit codes.
- **Immutable data** — Project coding style requires no mutation. CI commits replace data.json atomically.

### Integration Points
- CI workflow must exist at `.github/workflows/` (directory does not exist yet — greenfield)
- CI calls `go run ./cmd/meters fetch` then `go run ./cmd/meters build` (or equivalent)
- Deploy artifact uploads root files + `site/` + `.nojekyll`
- CI commit pushes to main branch on success (conditional on data.json change)
- `.nojekyll` must exist in artifact (and committed to repo)
- GitHub Pages source must be set to "GitHub Actions" in repo Settings (one-time manual step)
